# AGENTS.md

## 一、文件用途
本文件用于让接手本项目的 Codex / 工程代理在开始工作前，快速理解项目目标、当前边界、工作规则和代码约束。

---

## 二、项目概述

### 1. 项目名称
协鑫集成经营计划部智能平台

### 2. 当前优先建设范围
当前优先聚焦 **物流一期**。

### 3. 当前物流一期核心范围
- 历史数据：2023–2025 年 Excel
- 正式系统数据：2026 年后 MySQL
- 当前以 **结构化查询** 为主，不是纯文档 RAG 优先
- 当前已具备：
  - 历史 Excel 导入 ETL
  - 2026+ 系统数据同步
  - 统一查询服务（已演进到 V10）
  - 自然语言解析与模板匹配
  - SQL 模板绑定
  - 结果解释、空结果分析
  - 前端查询页 V1

### 4. 当前系统目标
在统一项目底座上，先完成物流域的：
- 自然语言查询
- 结构化查询
- 明细查询
- 跨来源对比查询
- 结果解释与日志追踪

后续才扩展：
- 物流文档知识库 / RAG
- 计划 BOM 域
- 经营分析域
- 物料管理域
- 工具层
- Agent 层

## 三、已确认的关键业务规则

### 1. 运量口径
- “运量”默认按 **瓦数口径**
- 标准指标编码：`shipment_watt`

### 2. 历史 / 系统来源规则
- 2023–2025：历史 Excel
- 2026 年后：正式系统数据
- 2026 年前系统库中的数据视为测试数据，不纳入正式统计

### 3. 混合查询规则
- 历史 Excel 与 2026+ 系统允许在上层统一语义层做混合查询
- 不允许直接在原始层粗暴混查

### 4. 当前仓库维度边界
当前按“路线 1”执行：
- **暂不补 allocate 链路**
- 仓库维度暂不作为一期可靠统计维度
- 不要把当前仓库字段的空值误判为同步 bug

## 四、当前代码状态判断规则

### 最重要规则
**不要根据历史聊天或历史补丁文件，推断当前仓库已经具备某项能力。**

一切必须遵循：
1. 先读取当前仓库代码
2. 先判断当前能力是否已真实合入
3. 再决定是否继续开发或修复

### 严禁行为
- 不要假设“之前做过的版本”一定已经在当前仓库里
- 不要用历史 zip 名称当事实来源
- 不要跳过代码审查，直接继续写下一版

## 五、Codex 工作流程要求

### 第一步：先审查再开发
每次开始工作时，必须先输出：
1. 当前仓库已完成能力判断
2. 当前未完成能力判断
3. 本次任务是否与当前仓库状态一致

### 第二步：复杂任务先给计划
如果任务涉及多个文件、多个步骤或多个工具，必须先给计划，再开始修改。

### 第三步：增量修改优先
- 优先增量修改
- 不大规模重构目录结构
- 不轻易改接口命名
- 不随意改变返回字段结构

### 第四步：所有新增和修改代码必须写中文注释
中文注释要求：
- 说明函数功能
- 说明参数含义
- 说明返回值
- 说明重要业务逻辑
- 对复杂判断、兼容逻辑、降级逻辑写清楚原因

### 第五步：完成后必须输出
- 修改文件清单
- 关键改动说明
- 测试方法
- 风险点
- 当前仍未解决的问题

## 六、代码风格要求

### 后端
- 保持当前 FastAPI / service / repository 分层风格
- 优先复用现有 service，不重复造一套新 service
- 保持当前 logistics 域目录结构不被破坏

### 前端
- 保持当前 Vue3 + Element Plus 风格
- 当前前端仍处在“可联调优先”的阶段
- 不要一上来做重 UI 重动画
- 优先保证：可查、可看、可解释、可回溯

### 命名
- 指标名、字段名、模板名与当前项目统一
- 若引入新命名，必须与现有语义兼容

## 七、安全与配置要求

### 1. 不要提交真实密钥
严禁提交：
- `.env`
- 真实数据库密码
- Redis / Milvus 密码
- 生产连接串
- API Key

### 2. 应只保留
- `.env.example`
- 示例配置
- 安全占位值

### 3. 打包与提交前应清理
以下文件不应进入最终交付包：
- `__pycache__/`
- `*.pyc`
- `__MACOSX/`
- `.idea/`
- `.pytest_cache/`
- `.env`
- `.DS_Store`

## 八、当前已知能力边界

### 1. 多域模板
虽然已经有：
- logistics
- plan_bom
- business_analysis
- material_management

但当前真正可执行能力仍然主要是 **logistics**。

### 2. SQL preview
当前 SQL preview 更偏调试与解释用途，不等同于数据库真实执行 SQL。

### 3. fallback
fallback 逻辑仅作为兼容兜底，不可输出误导业务的语义结果。
特别是：
- 不允许把“基地”冒充成“承运商/物流公司”

## 十、交付原则

当前阶段的目标，不是再无限叠加后端版本号，而是：
1. 让后端能力真正可被前端消费
2. 让业务看到可操作页面
3. 让系统具备试运行与演示条件

如果在“继续优化后端”和“完成前端可用性”之间有冲突，优先保证：
**前端可用性与联调闭环。**
