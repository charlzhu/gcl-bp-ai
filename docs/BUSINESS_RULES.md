# BUSINESS_RULES.md

## 一、物流一期核心业务规则

### 1. 数据来源规则
- 2023–2025：历史 Excel
- 2026 年后：正式系统 MySQL 数据
- 2026 年前系统数据视为测试数据，不纳入正式统计与正式分析

### 2. 查询路线规则
物流一期当前优先采用：
- 结构化查询
- 自然语言 -> 查询计划 -> SQL / QueryService

当前不是纯文档 RAG 优先。

### 3. 混合查询规则
历史 Excel 与 2026+ 系统数据允许在上层统一语义下混合查询。  
例如：
- 2025 年 3 月和 2026 年 3 月运量对比

底层需要分源处理，上层可以统一返回。

---

## 二、指标口径规则

### 1. 运量默认口径
“运量”默认按瓦数口径。
标准指标：
- `shipment_watt`

### 2. 业务同义表达
以下业务表达默认统一为运量（瓦数口径）：
- 运量
- 发货量
- 出货量

### 3. 系统正式运量口径
正式系统数据中，运量默认口径按：
- `logistic_ship_product.power * logistic_assign_detail.quantity`

### 4. 历史与系统统一展示规则
历史 Excel 与 2026+ 系统两个来源，在“运量”维度上按同一业务定义统一展示。

---

## 三、来源范围规则

### 1. 来源范围枚举
- `hist`：历史 Excel
- `sys`：2026+ 正式系统数据
- `all`：历史 + 系统混合范围

### 2. 默认来源推断规则
- 全部时间 <= 2025：默认 `hist`
- 全部时间 >= 2026：默认 `sys`
- 同时跨 2025/2026：默认 `all`

### 3. 显式来源优先
如果用户问题中显式出现：
- 历史台账 / 历史数据 -> `hist`
- 正式系统 / 系统数据 -> `sys`
则优先采用显式来源。

---

## 四、维度规则

### 1. 物流公司/承运商规则
以下表达统一理解为“物流公司”维度：
- 承运商
- 物流公司
- 物流承运商

标准字段：
- `logistics_company_name`

### 2. 区域规则
区域相关查询使用：
- `region_name`

### 3. 运输方式规则
以下表达需要标准化：
- 铁运 -> 铁路
- 公路 / 汽运 -> 同一标准值（以当前枚举配置为准）

标准字段：
- `transport_mode`

### 4. 仓库维度规则（非常重要）
当前物流一期采用路线 1：**不补 allocate 链路**。  
因此：
- 仓库维度不作为一期可靠统计维度
- 不要求 ship_task / assign_task 层的 warehouse 字段完整可用
- 与仓库强绑定的统计和分析暂不纳入一期可靠能力

---

## 五、明细查询规则

### 1. 明细查询识别
以下问题通常应走 `detail`：
- 合同编号 XXX 的明细
- SAP 单号 XXX 的明细
- 发货指令 XXX 的明细
- 询比价编号 XXX 的明细

### 2. 明细查询优先字段
可识别字段包括：
- `contract_no`
- `inquiry_no`
- `ship_instruction_no`
- `sap_order_no`
- `task_id`
- `vehicle_no`

### 3. 空结果解释规则
当 detail 查询为空时，应尽量区分：
- 该业务编号在中间层中不存在
- 过滤条件过严
- 当前来源范围不匹配

---

## 六、Fallback 规则

### 1. Fallback 只是兼容兜底，不代表正式结果
如果查询服务返回：
- `execution_mode = fallback`
则该结果优先作为兼容兜底解释，不宜视为正式可靠分析结果。

### 2. 物流公司字段不能在 fallback 中误用基地字段
当前 demo / fallback 数据中，若 `company` 实际表示基地，则：
- 不允许将其直接当成 `logistics_company_name`
- 不允许把“基地”冒充成“承运商/物流公司”返回给业务

---

## 七、多域规则

### 1. 当前真正可执行域
当前真正可执行的仍然主要是：
- `logistics`

### 2. 其他域当前状态
- `plan_bom`
- `business_analysis`
- `material_management`

当前更多是：
- 模板分层
- 路由能力
- 配置预留

不能假设这些域已经具备完整可执行查询能力。

---

## 八、代码开发规则（与业务强相关）

1. 不要假设历史补丁已经合入，一切以当前仓库为准。  
2. 新增和修改代码必须写中文注释。  
3. 复杂业务逻辑必须补注释说明：
   - 为什么这么做
   - 业务口径是什么
   - 边界在哪里
4. 任何涉及运量、来源范围、物流公司维度、fallback 的改动，都必须先检查本文件中的业务规则是否一致。
